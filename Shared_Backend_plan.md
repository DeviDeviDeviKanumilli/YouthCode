# EcoSentinel Shared Backend Plan

Use this file as the detailed milestone checklist for building the EcoSentinel backend. Use `AGENTS.md` as the compact operating guide for product rules, wording, architecture boundaries, and quality gates.

## How To Read This Plan

- Build milestones in order. Do not move to the next milestone until the current milestone's tests and acceptance criteria pass.
- Keep the backend API-first. The Expo app and research dashboard must call backend APIs instead of implementing ecological logic.
- Treat uncertainty, provenance, verification status, and privacy as core requirements.
- Prefer deterministic MVP providers and seeded tri-state data before adding live external integrations.
- Every new endpoint needs request/response schemas, OpenAPI tags, integration tests, and clear error responses.
- Every ecological service needs unit tests and plain-language explanation behavior where user-facing.

## Vertical Slice Target

The first convincing end-to-end backend demo should prove this path:

```txt
create user
-> create observation
-> attach media metadata
-> generate possible identification
-> normalize species if possible
-> enrich environmental context
-> compute Ecological Signal Priority
-> fetch Sighting Intelligence Card
-> fetch public Forecast Map layer
-> fetch research verification queue

```

The detailed milestones below break this vertical slice into safe implementation steps.

## Source Product Summary

EcoSentinel is an AI ecological intelligence platform. It is not a basic species identification app. It turns public nature sightings into structured, uncertainty-aware ecological signals that can be used by both everyday users and researchers.

The product has two frontend experiences:

1. Consumer Experience
- Built separately in Expo + React Native.
- Used by students, residents, hikers, teachers, and field observers.
- Main actions: upload photo, submit location, answer habitat questions, receive a Sighting Intelligence Card, view a simplified Forecast Map, and ask plain-language ecological questions.

2. Research Experience
- Built separately as a web dashboard.
- It must connect directly to this backend, independently of the React Native app.
- Used by researchers, conservation groups, park managers, teachers, and expert reviewers.
- Main actions: review sightings, verify/reject records, analyze map layers, inspect sampling gaps, filter observations, export research-ready data, and ask research-grade questions.

The backend owns all ecological intelligence.

The mobile app and research dashboard must not implement core ecological logic. They should call backend APIs.

Core backend responsibilities:
- User/auth support
- Observation creation
- Photo/media storage metadata
- Species candidate identification interface
- Taxonomic normalization
- Environmental enrichment
- Ecological Signal Priority scoring
- Forecast layer generation
- Sampling gap logic
- Verification workflow
- Research filters
- CSV/GeoJSON export
- AI assistant context API
- Strict uncertainty and provenance handling

Use an API-first architecture.

Recommended backend stack:
- Python 3.12+
- FastAPI
- PostgreSQL
- PostGIS
- SQLAlchemy 2.0 or SQLModel
- Alembic migrations
- Pydantic v2
- Redis for queues/caching where needed
- RQ, Celery, or Arq for background jobs
- S3-compatible storage interface for images; local storage adapter for development
- Pytest for tests
- Ruff for linting
- Mypy or Pyright for type checking
- Docker Compose for local services
- OpenAPI docs generated by FastAPI
- Optional later: GeoPandas, Shapely, Rasterio, PyProj, OSMnx, DuckDB, Pandas

Local development services:
- api
- postgres with PostGIS enabled
- redis
- optional minio for local S3-compatible storage

Coding requirements:
- Every endpoint must have Pydantic request/response schemas.
- Every database change must have an Alembic migration.
- Every service must have unit tests.
- Every public API endpoint must have integration tests.
- Every milestone must pass tests before moving to the next milestone.
- Do not hardcode secrets.
- Use environment variables with documented .env.example.
- Use structured logging.
- Use clear error responses.
- Use OpenAPI tags.
- Prefer explicit, typed service classes over large route handlers.
- Keep route handlers thin.
- Keep ecological logic in services, not controllers.
- Keep database queries in repositories where useful.
- Never let the LLM invent ecological facts.
- Never claim final species confirmation unless verification status supports it.
- Use words like “possible,” “likely,” “needs verification,” “high-value signal,” and “insufficient evidence.”
- Do not use alarmist labels like “danger score.”
- The main score is called Ecological Signal Priority.

Important product language:
- Correct: “Possible invasive concern.”
- Correct: “Needs expert verification.”
- Correct: “High-value verification candidate.”
- Correct: “Potential spread corridor.”
- Incorrect: “Confirmed invasion.”
- Incorrect: “Guaranteed prediction.”
- Incorrect: “Danger level.”
- Incorrect: “This species will spread here.”

M6 — Ecological Signal Priority Scoring
============================================================

Goal:
Build the transparent scoring engine that turns observations into ecological signal priorities.

M6.1 — Scoring model definition

Build:
Scoring constants:
- identity_confidence: weight 0.20
- local_novelty: weight 0.15
- habitat_match: weight 0.15
- pathway_risk: weight 0.15
- nearby_verified_record_context: weight 0.10
- ecological_sensitivity: weight 0.10
- sampling_gap_value: weight 0.10
- temporal_cluster_score: weight 0.05
- uncertainty_penalty: subtractive

Final score:
0–25: Low signal
26–50: Moderate signal
51–75: High-value verification candidate
76–100: Priority ecological signal

Special rule:
If evidence is too weak, label = insufficient_evidence even if numeric score exists.

Test:
- Formula deterministic.
- Weights sum correctly before penalty.
- Label thresholds correct.
- Uncertainty penalty cannot make score below 0.
- Score cannot exceed 100.

Acceptance:
- Core scoring math is stable.

M6.2 — Component scoring

Build component functions:

score_identity_confidence:
- Based on latest AI confidence.

score_local_novelty:
- Higher if few nearby records or possible range-edge signal.
- Lower if common locally.

score_habitat_match:
- Based on habitat answers and environmental context.

score_pathway_risk:
- Higher near waterway, road, trail, disturbed area, or relevant pathway.

score_nearby_verified_record_context:
- Higher if similar verified records exist nearby.

score_ecological_sensitivity:
- Higher near parks, wetlands, waterways, or sensitive habitats.

score_sampling_gap_value:
- Higher if area is under-sampled.

score_temporal_cluster:
- Higher if recent records cluster nearby.

score_uncertainty_penalty:
- Higher penalty for low confidence, poor media, coordinate uncertainty, unresolved species, missing context.

Test:
- Each function has unit tests.
- Missing inputs handled.
- Explanation reasons generated.
- No component silently crashes.

Acceptance:
- Score is explainable, not just numeric.
