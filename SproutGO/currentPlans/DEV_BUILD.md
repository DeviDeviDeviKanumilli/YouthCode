# DEV_BUILD.md — Running SproutGo on a device/simulator

**Why this doc exists:** SproutGo cannot run in **Expo Go**. The Map tab imports
`@rnmapbox/maps`, which ships its own native code that is not bundled into the Expo
Go app and cannot be loaded at runtime. You must build a **custom dev build** (your
own Expo Go-equivalent that includes Mapbox's native code). This is TECH_RISKS R1.

> Correction to earlier notes: `expo-camera` and `expo-location` DO run in Expo Go —
> they're part of the Expo SDK. `@rnmapbox/maps` is the sole reason a dev build is
> required.

There are two ways to produce that build. **Local (Xcode) is recommended** for this
machine — the toolchain is already installed.

---

## Prerequisites (all routes)

### Mapbox tokens — required, get them first
`@rnmapbox/maps` needs **two** tokens from a free account at
https://account.mapbox.com:

1. **Secret download token** — scope `DOWNLOADS:READ`, starts with `sk.`. Used at
   BUILD time to fetch the Mapbox SDK from Mapbox's private registry. Without it the
   iOS `pod install` / Android Gradle step fails. Env var: `MAPBOX_DOWNLOAD_TOKEN`.
2. **Public token** — starts with `pk.`. Used at RUNTIME by the map. Restrict it by
   bundle id (`com.sproutgo.app`). Env var: `EXPO_PUBLIC_MAPBOX_TOKEN`.

Put both in `apps/mobile/.env` (gitignored; copy from the repo-root `.env.example`),
plus the Supabase + API values the app already needs:
```
MAPBOX_DOWNLOAD_TOKEN=sk....
EXPO_PUBLIC_MAPBOX_TOKEN=pk....
EXPO_PUBLIC_SUPABASE_URL=...
EXPO_PUBLIC_SUPABASE_ANON_KEY=...
EXPO_PUBLIC_API_BASE_URL=http://localhost:3000   # or your Vercel URL
```

## Route A — Local build via Xcode (recommended on this Mac)

**Already installed here:** macOS 26, Xcode 26.3, CocoaPods 1.16.2, Node. Nothing
else needed for the iOS Simulator. No Expo account, no Apple Developer account.

From `apps/mobile`:
```bash
# 1. Load the tokens into your shell (or rely on .env via direnv/expo)
export MAPBOX_DOWNLOAD_TOKEN=sk....

# 2. Generate the native iOS project (runs config plugins: rnmapbox, location, router)
npm run prebuild:ios          # = expo prebuild --platform ios --clean

# 3. Compile + install + launch in the iOS Simulator (pod install happens here)
npm run ios                   # = expo run:ios
```
After the first build you can open `apps/mobile/ios/SproutGo.xcworkspace` in Xcode
and Run from there if you prefer the GUI (pick a Simulator, ▶). Day to day you only
re-run `npm start` (Metro) and reload JS — you do NOT rebuild unless you add/change a
native module.

**Physical iPhone** (optional): needs a free Apple ID signing team set in Xcode
(Signing & Capabilities → select your team), then Run with the device selected.
Free accounts work for personal-device development; the $99/yr Apple Developer
Program is only needed for TestFlight/App Store distribution.

Two physical-device gotchas (the simulator hides both):
- **API URL** — `EXPO_PUBLIC_API_BASE_URL` must NOT be `localhost` (that resolves to the
  phone itself). Use your deployed **Vercel HTTPS URL**, or your Mac's **LAN IP** over http
  (`http://192.168.x.x:3000`) while running `npm run api:dev`. The app already permits
  local-network cleartext (`NSAllowsLocalNetworking`), so the LAN-IP case works without
  extra setup; arbitrary internet http is still blocked.
- **Bundle-id signing collision** — if `com.sproutgo.app` is already claimed under the Apple
  ID you sign with, set `IOS_BUNDLE_ID=com.yourname.sproutgo` in `apps/mobile/.env` (then
  re-run `prebuild:ios`) instead of editing `app.config.ts`. If you do, restrict your
  `EXPO_PUBLIC_MAPBOX_TOKEN` to that same bundle id.

The generated `ios/` (and `android/`) folders are **gitignored** — they're build
output, regenerated by `prebuild`. The source of truth is `app.config.ts`.

## Route B — EAS cloud build

Use when you don't want to build locally or need to share an installable build.
Requires a free Expo account; `eas.json` is already in `apps/mobile`.
```bash
npm i -g eas-cli
eas login
eas init                      # fills extra.eas.projectId in app.config.ts
# store the build-time secret with EAS instead of a local env var:
eas secret:create --name MAPBOX_DOWNLOAD_TOKEN --value sk....
eas build --profile development --platform ios     # simulator build, no Apple acct
```
The `development` profile in `eas.json` is a dev-client + simulator build. A
physical-device iOS build through EAS DOES require the paid Apple Developer account
(for provisioning/signing). `preview`/`production` profiles are scaffolded for later.

## Android (not viable on this machine yet)

Local Android build is currently **blocked**: Java is 1.8 (Android Gradle needs
**JDK 17**) and `ANDROID_HOME` is unset (no Android SDK). To enable later: install
Temurin JDK 17 + Android Studio (which provides the SDK + an emulator), set
`ANDROID_HOME`, then `npm run android`. EAS Route B builds Android in the cloud and
sidesteps all of this. **Do iOS first.**

## Troubleshooting

- **`Failed to download Mapbox SDK` / 401 during pod install or Gradle** — the
  `MAPBOX_DOWNLOAD_TOKEN` is missing, lacks the `DOWNLOADS:READ` scope, or expired.
- **Map renders blank / "Unauthorized"** — `EXPO_PUBLIC_MAPBOX_TOKEN` is missing or
  URL-restricted to a different bundle id than `com.sproutgo.app`.
- **`@rnmapbox/maps` native module not found / app crashes on the Map tab** — you're
  running in Expo Go. Use the dev build (`npm run ios`), not `expo start` alone.
- **Node v26 engine warnings** — Expo SDK 51 targets Node 18/20. Warnings are
  usually harmless; if `prebuild` fails oddly, switch to Node 20 LTS via `nvm`.
- **Stale native build after adding a module** — re-run `npm run prebuild:ios`
  (`--clean` wipes and regenerates `ios/`), then `npm run ios`.

> Once a dev build runs, the full discovery loop (M1) + the map (M2) can finally be
> exercised end-to-end against live Supabase Storage + the bbox endpoint — the part
> that has only been unit-tested so far.
